# design-to-impl: 設計から実装までのマルチエージェントワークフロー

あなた (メインセッション) は**オーケストレーター**。計画の維持、タスクの委譲、ナレッジの管理、成果物の品質ゲートに責任を持つ。実作業は自分でやってもよいが、大きな調査・執筆・実装はサブエージェントに委譲する。

## 前提 (ワークスペース構成)

- ワークスペース直下の `CLAUDE.md` に、docs リポジトリ (設計成果物の配置先) と実装リポジトリの一覧がある
- `workflow/PLAN.md` = 計画と進捗、`workflow/KNOWLEDGE.md` = 共有ナレッジ。全エージェントは「開始前に読む・完了時に追記する」プロトコルに従う
- 利用可能なエージェント: `investigator` (読み取り専用調査), `doc-author` (docs 執筆), `design-reviewer` (設計レビュー), `service-implementer` (実装)。定義がなければ general-purpose で代替し、各定義ファイル相当の指示をプロンプトに含める

## 開始・再開時の手順

1. `workflow/PLAN.md` と `workflow/KNOWLEDGE.md` を読む
2. 新規テーマなら PLAN.md のテンプレートを埋める。テーマの目的・対象リポジトリが不明なら、ここでだけユーザーに確認する
3. TaskCreate で現在フェーズのタスクを登録し、進捗を PLAN.md と同期して管理する

## フェーズ定義

### 1. 調査・設計検討
- 既存の関連ドキュメント (docs リポジトリの該当 topic / ADR) と現状実装を把握する
- 独立した調査は `investigator` に並列委譲する。結果の「KNOWLEDGE.md 追記候補」を検討し、採用分を KNOWLEDGE.md へ追記する
- 設計上の主要な選択肢と論点を洗い出し、PLAN.md に記録する

### 2. 設計成果物の作成
- `doc-author` に委譲する。1 回の委譲 = 1 ドキュメント (または密結合な一組) を目安にする
- 複数選択肢の比較を伴う意思決定は ADR、構造・方針は design、調査・検証結果は report に分ける (docs リポジトリのルールに従う)
- ユーザーが判断すべき意思決定 (コスト、運用方針等のトレードオフ) は、ADR 化の前に AskUserQuestion で確認する

### 3. 設計レビュー
- `design-reviewer` に委譲し、レビューレポートを作らせる
- blocker / major 指摘は `doc-author` への改稿委譲 or 自分で修正して解消する。解消の判断が新たな意思決定になるなら ADR に反映する
- ゲート: doc-linter 通過 + blocker/major ゼロでフェーズ完了

### 4. 実装計画
- 設計をリポジトリ別・依存順のタスクに分解し、PLAN.md のタスクリストに書く
- 各タスクに「対象リポジトリ / 変更内容 / 完了条件 (テスト含む)」を明記する。この粒度が委譲プロンプトの品質を決める
- リポジトリ間の依存 (API 契約の変更順序等) を明示する

### 5. 実装 + テスト
- タスク単位で `service-implementer` に委譲する。委譲プロンプトには必ず「設計ドキュメントのパス / タスクの完了条件 / KNOWLEDGE.md プロトコル」を含める
- 同一リポジトリ内で並列させる場合は `isolation: "worktree"` を使う。異なるリポジトリなら worktree なしで並列してよい
- 実装から設計へのフィードバック (設計の穴) が報告されたら、実装を止めて設計側を修正するか判断し、KNOWLEDGE.md と該当ドキュメントに反映する

### 6. コードレビュー・仕上げ
- 各リポジトリの差分に `/code-review` を実行し、指摘を修正する
- 動作確認が可能なら `/verify` 相当の end-to-end 確認を行う
- コミット・PR 作成はユーザーの指示があった場合のみ

### 7. ナレッジ昇格
- KNOWLEDGE.md を清算する: 恒久的な知見は docs リポジトリの `knowledge/` (技術領域別) へ、運用手順は `runbooks/` へ、障害対応は `playbooks/` へ昇格する
- 昇格済み・不要になったエントリを KNOWLEDGE.md から削除し、PLAN.md を完了状態にする

## 委譲プロンプトの型

すべての委譲プロンプトに以下を含める:

```
## タスク
(ゴールと完了条件。曖昧さを残さない)

## 文脈
- まず workflow/KNOWLEDGE.md と workflow/PLAN.md を読むこと
- 関連ドキュメント: (パス列挙)
- 前提となる決定事項: (要点をインライン展開。エージェントは会話履歴を持たない)

## 報告に含めること
(成果物パス、判断とその根拠、KNOWLEDGE.md への追記内容 or 追記候補、残課題)
```

## オーケストレーターの規律

- **委譲の損益分岐**: サブエージェントはコールドスタートする。文脈説明のコスト < 並列化・コンテキスト節約の利益、のときだけ委譲する。数分で終わる小タスクは自分でやる
- **報告の検証**: エージェントの完了報告を鵜呑みにせず、成果物 (ファイル・テスト結果) を自分で確認してからフェーズを進める
- **継続対話**: 手戻りは新規スポーンではなく SendMessage で同じエージェントに差し戻す (文脈が残っている)
- **PLAN.md の鮮度**: フェーズ・タスクの状態変化のたびに PLAN.md を更新する。次セッションの自分が読者
